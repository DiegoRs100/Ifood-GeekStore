• Case
    Resumo

    O teste consiste na criação de um website composto por uma tela de login (autenticando via api) e um CRUD de produtos.

    As informações de acesso à API de login serão enviadas momentos antes de iniciar o teste.

    As ferramentas e tecnologias necessárias para realização do teste:

    • Sql Server 2014 ou superior
    • C# .net framework 4.5 ou .net core;
    • EntityFramework 6 ou core;
    • Html, Css e javascript;
    • Angular 6+;
    Login

    A tela de autenticação deve ser apenas composta pelos campos login e senha. Quando o usuário informar esses dados você deverá verificar se os dados são válidos realizando um post para a API "https://dev.sitemercado.com.br/api/login" e o retorno deverá ser tratado conforme o JSON na imagem abaixo. Se o usário e senha forem válidos, você deverá redirecionar o usuário para a tela de cadastro de produtos, caso contrário, retornar a mensagem retornada pela API para o usuário.

    API de login:

    • Usuário: 11234567890
    • Senha: 09876543211
    • Verbo: POST
    • Encoding: UTF8
    • Autenticação: HTTP Basic Authentication

    Produtos

    Na funcionalidade de produtos, você deverá desenvolver todas as funções de manutenção do cadastro (CRUD), inserção, edição, exclusão e listagem. Para o cadastro do produto apenas os seguintes campos são necessários: Nome do Produto, Valor de Venda e Imagem.

    Você deverá criar um banco de dados SQL Server com uma tabela chamada produto, composta pelos campos necessários para atender a tela.
  • Solução
    Arquitetura

    Foi criado uma API com arquitetura de 3 camadas, levando em consideração que o projeto GeekStore.Core está no mesmo nível do projeto GeekStore.Business, com a diferença que esse projeto processa apenas os serviços da própria arquitetura, fazendo assim com que o projeto GeekStore.Business seja exclusivo para o processamento das regras de negócios.

    A persistência no banco de dados foi feita utilizando o entity framework, e o banco pode ser recriado usando a funcionalidade de Migrations.

    Login

    Foi identificado que o link fornecido no case representaria um risco a segurança da aplicação caso fosse consumido diretamente da aplicação em Angular, pois ao retornar apenas um boleano, quaquer pessoa mal intencionada poderia facilmente simular a resposta do serviço de login para se logar na aplicação. Por conta disso o serviço de login foi encapsulado dentro da API do projeto GeekStore que trabalha devolvendo para o front-end um Token no formato JWT.

    Produtos

    Por uma questão de boas práticas e performace, optei por não realizar o salvamento das imagens diretamente no banco de dados. Como alternativa, foi criada a simulação do que seria um FileServer que armazena as imagens diretamente no disco e grava no banco de dados apenas a referência do arquivo.

    Vale resaltar que em uma situação real, o mais adequado seria armazenar as imagens em um CDN, um serviço do tipo blob como o oferecido pela Azure ou até mesmo um FileServer dedicado com todas as ferramentas de conformidade e segurança disponíveis.
